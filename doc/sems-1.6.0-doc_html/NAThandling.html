NAT handling

There is several issues to consider if NAT is involved, depending on the usage scenario and whether SEMS itself and/or the UAs that SEMS communicates with are behind NAT.

SEMS server behind NAT

In some deployment scenarios, SEMS itself is behind NAT, e.g. when deploying on AWS/EC2. If SEMS acts as a server accepting calls, and no other NAT handling is in place, this works only if it's a static NAT: In sems.conf (SEMS core configuration parameters) set the public_ip interface option to the public IP address, which is mapped by the NAT to the private IP address assigned to the server.

SEMS server on public IP, acting as SBC

If SEMS is used as SBC to separate the public network with UAs from the core SIP network, a few SBC profile options enable options that help to send SIP and RTP to the proper address:

  • dlg_nat_handling
  • enable_rtprelay
  • rtprelay_force_symmetric_rtp

See the sbc module documentation (SBC (flexible B2BUA) Application), especially the Readme file for more details.

SEMS server on public IP, acting as media server, clients behind NAT

In order to properly reach clients behind NAT, a few options in sems.conf (SEMS core configuration parameters) are important:

  • sip_nat_handling
  • force_symmetric_rtp
